# AGENTS.md

## Назначение

Этот репозиторий содержит статический интерактивный карьерный кейс
«Незабываемая сумка».

Codex реализует уже утверждённое решение. Он не определяет
самостоятельно концепцию, факты, тексты, визуальную драматургию или
профессиональное позиционирование автора.

## Источники истины

Перед любыми изменениями прочитать в следующем порядке:

1.  `docs/project-brief.md`
2.  `docs/storyboard.md`
3.  `docs/content.md`
4.  `docs/visual-spec.md`
5.  файл текущей задачи в `docs/codex-tasks/`

При противоречии:

1.  остановить изменение спорного участка;
2.  указать конфликт в отчёте;
3.  не выбирать самостоятельно новую формулировку или дизайн.

Приоритет документов:

1.  текущая утверждённая задача;
2.  `docs/storyboard.md`;
3.  `docs/content.md`;
4.  `docs/visual-spec.md`;
5.  `docs/project-brief.md`;
6.  `README.md`.

Текущая задача может уточнять способ реализации, но не должна менять
утверждённые факты и концепцию.

## Концепция

Центральная цепочка страницы:

**действие покупателя → кассовое событие → обработка → бонус → отчёт →
CRM-признак**

Основной маршрут состоит из семи разделов:

1.  вход;
2.  проблема;
3.  сквозная механика;
4.  роль автора;
5.  результаты;
6.  ограничения;
7.  карьерный вывод.

Порядок разделов менять нельзя.

## Технические ограничения

Использовать:

- семантический HTML;
- обычный CSS;
- vanilla JavaScript;
- локальные изображения;
- относительные пути;
- GitHub Pages.

Не добавлять без отдельного задания:

- React;
- Vue;
- Svelte;
- Astro;
- Next.js;
- npm;
- bundler;
- CSS framework;
- animation library;
- external component library;
- analytics;
- backend;
- CMS.

Если для конкретной задачи кажется необходимой новая зависимость:

1.  не устанавливать её;
2.  описать причину в отчёте;
3.  предложить вариант без зависимости.

## Правила работы с файлами

Изменять только файлы, явно разрешённые текущей задачей.

Не создавать дополнительные файлы без необходимости.

Не переименовывать существующие файлы и папки без отдельного задания.

Не удалять:

- документы из `docs/`;
- исходные изображения;
- реестр материалов;
- `.nojekyll`;
- утверждённые комментарии о границах доказательств.

Не изменять документы-источники в `docs/`, если текущая задача не
разрешает это явно.

## Контентные ограничения

Все публичные тексты должны соответствовать `docs/content.md`.

Нельзя:

- менять числа;
- округлять метрики иначе;
- добавлять новые результаты;
- усиливать причинно-следственные утверждения;
- утверждать доказанное сокращение пакетов;
- утверждать доказанный рост продаж или NPS;
- утверждать эффект относительно контрольной группы;
- использовать показатели масштабирования 2022 года;
- приписывать автору программирование SQL или кассовой системы;
- приписывать автору создание Wix-лендинга;
- скрывать ограничения проекта.

Разрешённые метрики:

- 2 630 покупателей;
- 24 589 покупок;
- 9 покупок в среднем;
- 59% транзакций Mastercard.

## Роль автора

Необходимо ясно разделять:

### Автор определил

- бизнес-задачу;
- пользовательский и кассовый сценарий;
- требования к бонусной логике;
- требования к транзакционному отчёту.

### Автор координировал и проверял

- требования к QR-бирке;
- взаимодействие с Mastercard;
- юридические ограничения;
- тестовые покупки;
- начисление бонусов.

### Команды реализовали

- производство физического продукта;
- кассовые изменения;
- серверную и SQL-логику;
- магазинные и покупательские материалы.

## Интерактив

На сайте предусмотрены только два основных последовательных
взаимодействия:

1.  покупательская кассовая цепочка;
2.  превращение транзакции в CRM-сигнал.

Основное содержание не должно зависеть от:

- клика;
- hover;
- drag;
- точного положения курсора;
- завершения анимации.

Первое прохождение шагов идёт в утверждённом порядке.

После завершения пользователь может вернуться к любому пройденному шагу.

Роль автора, метрики и ограничения должны быть полностью понятны без
взаимодействия.

## Работа с изображениями

Использовать только локальные файлы из `assets/`.

Запрещено:

- использовать временные Notion CDN URL;
- загружать внешние изображения по URL;
- заменять материалы стоковыми изображениями;
- использовать сгенерированные изображения;
- создавать вымышленный интерфейс CRM;
- публиковать исходный отчёт с идентификаторами.

Исходники находятся в:

`assets/images/original/`

Подготовленные версии находятся в:

`assets/images/web/`

В HTML использовать относительные пути:

`assets/images/web/file-name.webp`

Не использовать пути, начинающиеся с `/`.

Не изменять исходные изображения в папке `original`.

Обработанные и обезличенные версии сохранять как отдельные файлы.

## Доступность

Каждая задача должна сохранять или улучшать:

- корректную иерархию заголовков;
- семантические области страницы;
- клавиатурную навигацию;
- видимый focus;
- достаточный контраст;
- альтернативные описания изображений;
- читаемость без JavaScript;
- понятность без hover;
- поддержку `prefers-reduced-motion`;
- минимальную активную область 44 × 44 px;
- отсутствие горизонтальной прокрутки.

Не использовать ARIA вместо корректного HTML, если доступен нативный
семантический элемент.

## Адаптивность

Основные контрольные ширины:

- телефон: 375 px;
- крупный телефон: 430 px;
- планшет: 768 px;
- ноутбук: 1280 px;
- широкий экран: 1440 px.

На телефоне:

- основной визуал не закрепляется на весь экран;
- содержание идёт в обычном вертикальном потоке;
- основной смысл не скрывается в аккордеонах;
- горизонтальная прокрутка не используется для чтения;
- прогресс может быть закреплён только компактной строкой.

## Визуальные ограничения

Сохранять:

- документальную подачу;
- реальные материалы;
- нейтральный фон;
- одну постоянную акцентную систему;
- различие между фактом и интерпретацией;
- сдержанное движение.

Не использовать:

- параллакс;
- автоматическую прокрутку;
- scroll hijacking;
- бесконечную анимацию;
- декоративные счётчики от нуля;
- тяжёлые тени;
- имитацию SaaS-панели;
- чрезмерные скругления;
- игровую стилистику;
- фирменные цвета брендов как общий фон сайта.

## Производительность

Предпочитать:

- WebP или AVIF для подготовленных изображений;
- фиксированные `width` и `height`;
- ленивую загрузку изображений ниже первого экрана;
- минимальное количество JavaScript;
- системные шрифты или явно утверждённые локальные web-fonts;
- отсутствие сторонних запросов.

Не оптимизировать изображения путём замены исходных материалов
реконструкциями.

## Проверка

После изменений обязательно проверить:

1.  страницу через локальный HTTP-сервер;
2.  отсутствие ошибок в консоли;
3.  все относительные пути;
4.  ширины 375, 768, 1280 и 1440 px;
5.  клавиатурную навигацию;
6.  режим уменьшенного движения;
7.  отсутствие горизонтального скролла;
8.  соответствие текстов `docs/content.md`;
9.  соответствие порядка экранов `docs/storyboard.md`;
10. отсутствие новых неподтверждённых утверждений.
