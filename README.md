# Незабываемая сумка — интерактивный карьерный кейс

Короткая документальная веб-история о том, как использование
многоразовой сумки в магазине было связано с кассовым событием,
программой лояльности, бонусной обработкой, транзакционным отчётом и
CRM-признаком покупателя.

## Статус проекта

Текущий этап: реализация утверждённого storyboard.

Источники истины для содержания и дизайна:

- `docs/project-brief.md`
- `docs/storyboard.md`
- `docs/content.md`
- `docs/visual-spec.md`

Изменения фактов, метрик, последовательности экранов и центральной
концепции допускаются только после отдельного согласования.

## Центральная идея

Страница показывает одну причинно-следственную цепочку:

**действие покупателя → кассовое событие → обработка → бонус → отчёт →
CRM-признак**

Основной интерактив разделён на две последовательные части:

1.  покупательская кассовая цепочка;
2.  превращение транзакции в CRM-сигнал.

Роль автора, метрики и ограничения должны быть понятны без кликов и
дополнительных раскрытий.

## Структура страницы

1.  Вход в кейс.
2.  Проблема.
3.  Сквозная механика.
4.  Роль автора.
5.  Результаты пилота.
6.  Ограничения.
7.  Карьерный вывод и ссылка на полное доказательное досье.

## Подтверждённые показатели

На странице разрешено использовать только следующие показатели
пятимесячного пилота:

- 2 630 покупателей вошли в механику через покупку QR-сумки;
- 24 589 покупок прошли со сканированием сумки;
- 9 покупок в среднем пришлось на одного привлечённого покупателя;
- 59% транзакций внутри механики были оплачены Mastercard.

Нельзя подавать как подтверждённые результаты:

- точное сокращение количества одноразовых пакетов;
- рост продаж экологичного ассортимента;
- рост частоты покупок относительно контрольной группы;
- рост NPS;
- реактивацию покупателей;
- показатели масштабирования 2022 года без отдельного подтверждающего
  источника.

## Роль автора

Автор:

- сформулировал бизнес-задачу;
- собрал пользовательский и кассовый сценарий;
- согласовал требования к сумке, QR-коду, бонусам и данным;
- подготовил требования к транзакционному отчёту;
- участвовал в юридическом согласовании;
- провёл тестовые покупки;
- проверил начисление бонусов;
- координировал взаимодействие с Mastercard.

Не следует утверждать, что автор лично:

- программировал SQL-процедуру;
- разрабатывал кассовое программное обеспечение;
- производил сумки;
- создавал дизайн или код покупательского Wix-лендинга;
- самостоятельно реализовал все компоненты проекта.

## Технический подход

Сайт реализуется без framework и сборщика:

- семантический HTML;
- обычный CSS;
- небольшой vanilla JavaScript;
- локальные изображения;
- GitHub Pages.

На первом этапе создаётся полностью читаемая статическая версия всех
разделов.

После этого отдельно добавляются:

1.  закреплённая визуальная область;
2.  последовательные состояния механики;
3.  возврат к пройденным шагам;
4.  режим уменьшенного движения;
5.  финальная доступность и оптимизация.

## Основные файлы

- `index.html` — структура и содержание страницы;
- `styles.css` — сетка, типографика, адаптивность и визуальные
  состояния;
- `script.js` — интерактивная логика;
- `AGENTS.md` — обязательные правила для Codex;
- `docs/` — утверждённые материалы и задачи;
- `assets/` — изображения, реестр материалов и альтернативные описания.

## Работа с изображениями

Временные ссылки Notion нельзя использовать в production-коде.

Исходные материалы хранятся в:

`assets/images/original/`

Подготовленные для сайта версии хранятся в:

`assets/images/web/`

Обезличенные версии материалов с данными должны использовать отдельное
имя с суффиксом `-redacted`.

На сайте разрешены только относительные пути, например:

`assets/images/web/01-hero-bag.webp`

## Локальный запуск

Из корня репозитория выполнить:

    python3 -m http.server 8000

После этого открыть:

    http://localhost:8000

Не проверять сайт через прямое открытие `index.html` как локального
файла. Локальный сервер точнее воспроизводит работу относительных путей
и поведение сайта после публикации.

## Проверка перед публикацией

Проверить:

- порядок всех семи разделов;
- точность текстов относительно `docs/content.md`;
- четыре разрешённые метрики;
- отсутствие неподтверждённых результатов;
- корректность относительных путей;
- отсутствие персональных данных;
- читаемость на телефоне;
- работу клавиатурной навигации;
- отсутствие горизонтальной прокрутки;
- режим уменьшенного движения;
- наличие альтернативных описаний изображений.

## Публикация через GitHub Pages

В настройках репозитория:

1.  открыть `Settings`;
2.  выбрать `Pages`;
3.  в разделе `Build and deployment` выбрать `Deploy from a branch`;
4.  выбрать ветку `main`;
5.  выбрать папку `/(root)`;
6.  сохранить настройки.

После публикации адрес сайта появится в разделе GitHub Pages.

Файл `.nojekyll` должен оставаться в корне репозитория.

## Доказательная база

Полный исходный карьерный кейс хранится отдельно в Notion.

Публичная страница не должна полностью дублировать досье. Она показывает
короткий маршрут, а ссылка в финальном разделе ведёт к подробным
материалам и доказательствам.

## Ограничения реализации

Нельзя:

- менять центральную концепцию;
- переставлять экраны;
- переписывать утверждённые тексты без задания;
- придумывать дополнительные результаты;
- добавлять framework или сборщик без отдельного обоснования;
- использовать стоковые или сгенерированные изображения вместо реальных
  материалов;
- создавать вымышленный экран CRM;
- использовать обязательные клики для доступа к основному содержанию;
- блокировать прокрутку страницы;
- делать интерактив необходимым для понимания роли автора, метрик или
  ограничений.
